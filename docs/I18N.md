# Internacionalização (i18n) — Guilds Comercial

Documento operacional. Como o app suporta usuários fora do Brasil e como continuar a migração.

## Estado atual (2026-04-28)

### ✅ Pronto

**Infra (camada A — UI textual)**
- Sistema de i18n caseiro em [`lib/i18n/index.ts`](../lib/i18n/index.ts) com cookie `NEXT_LOCALE`
- 2 locales: `pt-BR` (default) e `en-US`

- Helpers `getServerLocale()` (RSC), `getClientLocale()` (browser), `setClientLocale()`, `getT(locale)` retorna função `t(key)`, `formatNumber/Date/Currency` com Intl

**Páginas migradas:**
- Sidebar (full)
- `/hoje` (cockpit completo)
- `/configuracoes` layout + tabs + perfil
- MobileNav, TrialBanner

**Domínio (camada B — fiscal/regional)**
- `organizacoes.pais` (ISO 3166-1, default 'BR')
- `organizacoes.tax_id` genérico (substitui CNPJ pra estrangeiros)
- `organizacoes.idioma_padrao` e `organizacoes.moeda_padrao`
- Constraint de CNPJ: condicional — só valida se `pais = 'BR'`
- Constraint de regime tributário: nullable se `pais != 'BR'`
- [`lib/utils/i18n-fiscal.ts`](../lib/utils/i18n-fiscal.ts) com:
  - `PAISES` (20 países top + "Outro")
  - `FUSOS_GLOBAIS` (~30 fusos por região)
  - `IDIOMAS`, `MOEDAS`
  - `validarTaxId(value, pais)` — delega validação por país (BR usa CNPJ por DV)
  - `labelTaxId(pais, locale)` — retorna "CNPJ" / "EIN" / "RFC" / "VAT" / etc
  - `formatTelefoneI18n` / `normalizarTelefoneI18n` / `isValidTelefoneI18n` via [libphonenumber-js](https://www.npmjs.com/package/libphonenumber-js)

**Onboarding internacional**
- Step 1 do wizard: select país (default detectado por país do user no futuro), tax_id label condicional
- Action `finalizarOnboarding` aceita `pais`, `tax_id`, `idioma_padrao`, `moeda_padrao`

**Configurações da organização**
- Form `/configuracoes/organizacao` com select país + idioma + moeda + fuso global
- Campos endereço condicional: BR usa CEP/UF, outros usam postal_code/região
- Tax ID com label dinâmico
- Regime tributário só aparece se BR

**IA (camada C)**
- Tabela `ai_prompts` ganhou coluna `idioma` (default 'pt-BR')
- Dispatcher seleciona prompt pelo `idioma_padrao` da org com fallback (org+idioma → org+pt-BR → global+idioma → global+pt-BR)
- Variável `{{idioma}}` injetada automaticamente nas vars do user_template
- System prompt recebe instrução automática: "Respond in en-US" / "Responda em pt-BR"
- Aplicado em ambos `invokeAI` e `invokeAISystem` (cron)

**Push notifications (camada C)**
- [`lib/push-i18n.ts`](../lib/push-i18n.ts) com `buildPushPayload(evento, locale, vars)` retornando `{title, body}` traduzido
- Templates por evento × locale: `cadencia_vencendo`, `resumo_diario`, `lead_fechado_proposta`, `lead_reabriu`
- Cache de `getOrgLocale(orgId)` por 5min pra cron evitar consultas repetidas
- Callers atualizados: `moverEtapa`, `daily-digest`, `push-cadencia`

**Emails (camada C)**
- `sendInviteEmail` e `sendWelcomeEmail` aceitam `locale` (pt-BR ou en-US)
- Onboarding passa `idioma_padrao` da org

**Stripe multi-currency (camada D)**
- Tipo `Currency` = `'BRL' | 'USD' | 'EUR'` em [`lib/stripe.ts`](../lib/stripe.ts)
- Env vars seguem padrão `STRIPE_PRICE_{PLAN}_{CURRENCY}` (ex: `STRIPE_PRICE_STARTER_USD`); fallback para `STRIPE_PRICE_{PLAN}` é tratado como BRL
- `getStripePriceId(plan, currency)`, `isStripeConfiguredForPlan(plan, currency)`, `planFromStripePrice` (busca em todas as moedas)
- `createCheckoutSession` aceita `currency` opcional (default BRL); inclui line_item de overage por moeda
- `lib/billing.ts`: PLANS agora têm `priceByCurrency` ao invés de `priceLabel`; helper `priceLabelOf(plan, currency)`
- `/configuracoes/billing` lê `moeda_padrao` da org e renderiza preços corretos

## Como continuar a migração

### Adicionando novas strings na UI

Em qualquer **client component**:

```tsx
"use client";
import { useEffect, useState } from "react";
import { getClientLocale, getT, type Locale } from "@/lib/i18n";

export default function MyComponent() {
  const [locale, setLocale] = useState<Locale>("pt-BR");
  useEffect(() => setLocale(getClientLocale()), []);
  const t = getT(locale);

  return <h1>{t("minha_secao.titulo")}</h1>;
}
```

Em **server component (RSC)**:

```tsx
import { getServerLocale, getT } from "@/lib/i18n";

export default async function MyPage() {
  const locale = await getServerLocale();
  const t = getT(locale);
  return <h1>{t("minha_secao.titulo")}</h1>;
}
```

E adicionar a chave em **ambos** `pt-BR.json` e `en-US.json`:

```json
{
  "minha_secao": {
    "titulo": "Meu título"
  }
}
```

Se a chave não existir, `t()` retorna o próprio path (visível no UI = fácil de detectar).

### Adicionando novo idioma (ex: es-ES)

1. Criar `lib/i18n/messages/es-ES.json` traduzindo de pt-BR.json
2. Em [`lib/i18n/index.ts`](../lib/i18n/index.ts) adicionar à `LOCALES` e ao map `MESSAGES`
3. Atualizar `IDIOMAS` em [`lib/utils/i18n-fiscal.ts`](../lib/utils/i18n-fiscal.ts)
4. Adicionar em [`components/i18n/locale-toggle.tsx`](../components/i18n/locale-toggle.tsx)

### Adicionando novo país

Em [`lib/utils/i18n-fiscal.ts`](../lib/utils/i18n-fiscal.ts) na constante `PAISES`:

```ts
{ code: "JP", nome_pt: "Japão", nome_en: "Japan", moeda_padrao: "USD", idioma_padrao: "en-US" },
```

E se quiser label específico de tax_id, adicionar em `labelTaxId()`:

```ts
if (pais === "JP") return "Corporate Number";
```

### Adicionando nova moeda no Stripe

1. Criar produtos + prices no Stripe Dashboard em USD/EUR/etc
2. Adicionar env vars `STRIPE_PRICE_STARTER_USD`, `STRIPE_PRICE_GROWTH_USD`, `STRIPE_PRICE_SCALE_USD` (e `STRIPE_PRICE_AI_OVERAGE_USD` para metered billing)
3. Adicionar moeda em `SUPPORTED_CURRENCIES` em [`lib/stripe.ts`](../lib/stripe.ts) se ainda não estiver
4. Atualizar `priceByCurrency` em [`lib/billing.ts`](../lib/billing.ts) com label da moeda nova

## Decisões técnicas

### Por que i18n caseiro e não next-intl?

Trade-off: next-intl tem features mais completas (pluralização, formato avançado, AST) mas requer URL prefix (`/pt-BR/hoje`) ou setup pesado pra evitar isso. Pra V1 com 2 idiomas e maior parte do app já funcionando, caseiro é mais leve (zero deps, ~80 linhas) e migrar pra next-intl depois é mecânico.

### Por que cookie e não URL prefix?

URL prefix duplica todas as rotas (`/pt-BR/hoje` vs `/en-US/hoje`) e o app é privado (atrás de login). Pra SEO de marketing público no futuro, podemos adicionar prefix só nas páginas marketing. Cookie é decisão de UX por usuário.

### Por que tax_id genérico em vez de country-specific validation?

V1 prioriza desbloqueio de cadastro internacional. Validação por país (CUIT, RUT, NIF, EIN) é incremental — adicione cada uma em `validarTaxId()` quando vier o cliente do país.

### Por que weighted units no Stripe metered ao invés de price por moeda + por feature?

Manter 1 price metered por moeda × 1 weighted unit = R$0,30/USD$0,06 etc é mais simples que 15 prices por feature. Cliente vê na fatura "1234 unidades de IA" e UI do Guilds mostra detalhamento por feature em `/configuracoes/billing`. Trade-off explícito em [`docs/AI_OVERAGE_BILLING.md`](./AI_OVERAGE_BILLING.md).
